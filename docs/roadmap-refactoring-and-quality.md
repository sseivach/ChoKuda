# Дорожная карта: рефакторинг и повышение качества

## Цель

Сделать ChoKuda проще в поддержке, безопаснее для дальнейшей разработки и устойчивее к ошибкам без изменения пользовательского поведения.

Главный фокус: постепенно разгрузить большой Razor-компонент, закрепить ключевые сценарии тестами, привести ошибки в UI к понятному виду и подготовить файловое хранилище к будущим изменениям формата.

## Принципы работы

- Не переписывать все сразу. Каждый шаг должен быть маленьким, проверяемым и откатываемым.
- Не менять поведение приложения во время рефакторинга без отдельного решения.
- Сначала фиксировать текущее поведение тестами и ручными проверками, потом переносить код.
- `Core` хранит правила предметной области и работу с данными.
- App-level VM/сервисы хранят состояние экрана, команды UI и orchestration между `Core` и Razor.
- Razor-компоненты должны отвечать за разметку, биндинги и простые UI-события.
- После каждого этапа приложение должно собираться, тесты должны проходить, ручной smoke-check должен быть выполнен.

## Текущее состояние

- Синхронизировано с фактическим состоянием проекта 2026-07-15.
- Основной экран стартует из `src/ChoKuda.App/Components/AppShell.razor`, крупные UI-блоки вынесены в дочерние Razor-компоненты.
- Состояние поиска, фильтров, точки, коллекции, вложений и операций вынесено в app-level VM/сервисы:
  - `SearchViewModel`;
  - `FilterViewModel`;
  - `LibraryDataViewModel`;
  - `PointEditorViewModel`;
  - `CollectionEditorViewModel`;
  - `AttachmentDraftState`;
  - `UiOperationViewModel`;
  - `LibraryCoordinator`;
  - `BootstrapIconCatalog`;
  - `AttachmentDropService`.
- `AppShell.razor` все еще содержит верхнеуровневую orchestration-логику:
  - работа с картой;
  - связывание дочерних компонентов с VM/сервисами;
  - вызовы `LibraryCoordinator`, `PointService`, `CollectionService` и сервисов вложений;
  - тонкие UI-wrapper'ы для confirm/dialog/window-сценариев;
  - загрузка данных после save/delete/import;
  - безопасный retry для library setup/reload.
- В `Core` уже есть полезные сервисы, которые нужно переиспользовать, а не дублировать:
  - `PointService`;
  - `CollectionService`;
  - `PointFilterService`;
  - `SearchService`;
  - сервисы вложений;
  - модели карты и документов.

## Не делаем в рамках этого roadmap

- Не меняем выбранный стек: WPF + BlazorWebView/WebView2.
- Не переносим приложение на web/mobile/cross-platform.
- Не меняем провайдера карт.
- Не добавляем базу данных.
- Не делаем enterprise-архитектуру с лишними слоями ради слоев.
- Не добавляем крупные пользовательские функции без отдельного решения.
- Малые уже утвержденные UX-доработки допускаются, если не ломают модель хранения и покрыты тестами.

## Этап 2. Вынос логики из Razor-компонента

### Цель

Сделать `AppShell.razor` тонким экраном, а состояние и команды перенести в понятные App-level VM/сервисы.

Важно: UI-состояние не уходит в `Core`. В `Core` остаются чистые правила и работа с данными. Состояние экрана, dirty-state, выбранные панели, результаты поиска и ошибки живут в приложении.

### Целевая схема

- `AppShell.razor`
  - общая компоновка экрана;
  - подключение дочерних компонентов;
  - биндинги;
  - вызов команд VM/сервисов.
- App-level VM/сервисы
  - состояние экрана;
  - команды пользователя;
  - координация между UI и `Core`;
  - подготовка данных для отображения.
- `Core`
  - документы точек и коллекций;
  - чтение и запись файлов библиотеки;
  - поиск;
  - фильтрация;
  - нормализация тегов;
  - правила primary collection;
  - импорт и классификация вложений.

## Этап 4. Файловое хранилище: одна папка и ручной backup

### Цель

Не усложнять локальное хранилище. Пользовательские данные ChoKuda живут в одной выбранной папке библиотеки, а backup делается вручную обычным копированием этой папки.

Важно: `%APPDATA%\ChoKuda\appsettings.json` остается технической настройкой приложения, где хранится путь к выбранной библиотеке и app-level параметры. Это не основная библиотека и не backup-цель. Ценная пользовательская библиотека - выбранная папка `ChoKudaLibrary`.
