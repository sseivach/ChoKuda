# AppShell Responsibility Map

Дата baseline: 2026-07-08.

Этот документ фиксирует текущее состояние `src/ChoKuda.App/Components/AppShell.razor` перед рефакторингом. Его задача - показать, какие сценарии сейчас сосредоточены в одном Razor-компоненте и в каком порядке их безопаснее выносить.

## Baseline-проверки

- `dotnet build ChoKuda.slnx` - успешно.
- `dotnet test ChoKuda.slnx --no-build` - успешно, 156/156.
- `dotnet test tests\ChoKuda.Core.Tests\ChoKuda.Core.Tests.csproj --no-build --collect:"XPlat Code Coverage"` - успешно.
- Coverage по Cobertura:
  - line-rate: 100%;
  - branch-rate: 100%;
  - lines: 939/939;
  - branches: 216/216.
- Полный ручной smoke-check по `docs/MANUAL_TESTING.md` не выполнялся в рамках автоматического baseline.

## Общий размер и риск

- `AppShell.razor` содержит около 1500 строк.
- Компонент смешивает:
  - layout;
  - UI-состояние;
  - команды пользователя;
  - orchestration сервисов;
  - маппинг данных для карты;
  - ошибки;
  - draft-состояние форм;
  - работу с вложениями.

Основной риск: при добавлении новых функций легко менять несколько несвязанных сценариев в одном файле и ломать поведение без явного сигнала.

## Карта зон ответственности

### 1. Внедрение сервисов и общий layout

Примерный диапазон: строки 1-18.

Ответственность:

- подключение `Core`-сервисов;
- подключение app-level сервисов WPF;
- подключение `IJSRuntime`;
- верхний контейнер экрана.

Кандидат на изменение:

- оставить в `AppShell.razor` только зависимости, реально нужные самому shell;
- зависимости дочерних сценариев перенести в VM/сервисы или дочерние компоненты.

### 2. Левая панель: поиск, результаты, коллекции, фильтры, теги

Примерный диапазон: строки 20-246.

Ответственность:

- collapse/expand левой панели;
- поиск;
- вывод результатов поиска;
- создание и редактирование коллекций;
- выбор иконки коллекции;
- выбор цвета коллекции;
- фильтр коллекций;
- drag-and-drop порядка выбранных коллекций;
- фильтр тегов.

Кандидаты на вынос:

- `SearchViewModel`;
- `FilterViewModel`;
- `CollectionEditorViewModel`;
- компоненты `SearchPanel`, `CollectionsPanel`, `TagsPanel`.

### 3. Карта и настройка Stadia API key

Примерный диапазон: строки 250-285.

Ответственность:

- показ карты при наличии Stadia key;
- экран настройки Stadia key;
- передача точек и временного маркера в `MapView`;
- обработка кликов по карте, пинам и `Esc`.

Кандидаты на вынос:

- `MapWorkspace` как компонент;
- app-level состояние выбранной координаты;
- команды открытия точки и начала создания точки.

### 4. Правая панель точки

Примерный диапазон: строки 288-470.

Ответственность:

- address/region как текстовое поле;

- привязка коллекций;
- выбор primary collection;
- фото;
- файлы;
- ошибки точки и вложений;

- `Delete`.

Кандидаты на вынос:

- `PointEditorViewModel`;

- компоненты `PointPanel`, `AttachmentPanel`.

### 5. Нижняя служебная панель

## Рекомендуемый порядок выноса

1. Поиск: самый маленький и изолированный кусок.
2. Фильтры: связаны с поиском и картой, но уже хорошо покрыты `Core`-сервисами.
3. Правая панель точки: самый важный пользовательский сценарий.
4. Вложения: высокая цена ошибки, лучше выносить после стабилизации point editor.
5. Коллекции: отдельный редактор, но влияет на точки и фильтры.
6. Разделение Razor-компонентов: делать после выноса состояния, чтобы не размазать текущую сложность по файлам.

## Инварианты, которые нельзя сломать

- Поиск запускается только кнопкой `Search`.
- Пустой или пробельный поиск неактивен.
- Очистка поля поиска не меняет карту до нажатия `Reset search` или `Search`.
- Фильтры коллекций и тегов независимы.
- Reset коллекций не сбрасывает теги и поиск.
- Reset тегов не сбрасывает коллекции и поиск.
- Клик по результату поиска центрирует карту без изменения zoom.
- Клик по карте создает временную координату, но не открывает панель сам по себе.
- `Add point` активен только при выбранной временной координате и закрытой правой панели.
- `Esc` сбрасывает временный маркер и координаты.
- Файлы копируются в библиотеку только при `Save`.
- Один плохой файл не отменяет хорошие.
- Удаление точки, коллекции и saved-вложения требует подтверждения.
- Если точка имеет коллекции, primary collection обязательна.
- При удалении primary collection из точки новая primary выбирается по алфавиту из оставшихся коллекций.
